# 中文注解与文档写作规范

## 目标

本仓库默认使用中文注解与中文仓库文档，目标不是“注释越多越好”，而是让读代码、跑实验、查报告的协作者都能快速理解：

- 这个模块负责什么
- 这个函数输入输出是什么
- 这里为什么这样设计
- 哪些约束不能被破坏

## 适用范围

- Python 模块级 docstring
- 类与公开函数 docstring
- 关键代码块注释
- README、设计说明、贡献文档
- 运行与报告产物中的固定说明文字

## 基本规则

- 默认使用 UTF-8。
- 默认使用中文书写注解；必要的技术名词可保留英文原词。
- 注解优先解释“职责、约束、意图”，不要重复代码表面行为。
- 新增注释应短而准，避免段落式堆砌。
- 面向用户的仓库说明与报告固定结构应保持中文一致性。

## Docstring 规范

推荐写法：

- 模块 docstring：一句话说明模块职责，再补 2 到 4 行关键背景或约束。
- 类 docstring：说明这个类表示什么、管理什么状态、服务哪条链路。
- 函数 docstring：优先说明“做什么”和“返回什么”；只有在必要时补失败模式或副作用。

不推荐写法：

- “给变量赋值”“遍历列表”这类低信息描述
- 把实现细节全部抄进 docstring
- 同一信息在注释和文档里重复三遍

## 行内注释规范

只有在下列情况才建议写行内或代码块注释：

- 存在非显然约束
- 有意做了权衡或保守处理
- 顺序、批量、缓存、限流、恢复逻辑容易被误改
- 需要帮助读者区分“机制本身”和“工程护栏”

## 术语建议

- `run`：运行、单次运行
- `report`：正式报告
- `figure`：图表、图资产
- `cache shard`：缓存分库
- `rate limiter`：限流器
- `settle / reconcile`：对账、回写真实 usage
- `canonical path`：规范落点、正式落点

技术术语不必强行全部翻译，但同一概念应尽量全仓统一。

## 仓库文档规范

- README 讲“项目是什么、怎么用、去哪里看”
- `docs/` 讲长期结构、流程和规范
- `src/*/README.md` 讲目录职责
- `runs/README.md` 讲归档策略，不讲实现细节
- `CONTRIBUTING.md` 讲提交流程、同步要求和写作约定

## 提交前检查

- 新增或重构共享模块时，检查 docstring 是否仍是乱码或残留英文旧壳
- 改 CLI、配置名或目录结构时，同步 README 和 `docs/`
- 文本 I/O 是否显式使用 `encoding="utf-8"`
- 术语是否与现有文档一致
