# 一、方法流程

你原来那套想法不推翻，只是**做一次“拆层”**。
正式版本建议改成：

**Stage 0：问题规约与预算协议**
**Stage 1：独立求解与结构化输出**
**Stage 2：触发判断**
**Stage 3：分歧对象抽取**
**Stage 4：预算感知路由**
**Stage 5：选择性通信**
**Stage 6：局部审计**
**Stage 7：最终聚合与失败保护**

这样改完以后，每个阶段都能单独 ablate，论文就会非常干净。

------

# 三、各阶段任务细化（方法流程层面）

## Stage 0：问题规约与预算协议

### 目标

在任何实验开始前，先把“你究竟在优化什么”固定下来。

### 必做任务

1. 定义论文的主目标：
   在固定总预算下最大化准确率，而不是只追求最高准确率。

2. 明确预算口径：
   预算必须是**总成本**，不能只算通信 token。建议统一为：
   $$
   B_{\text{total}} = B_{\text{solve}} + B_{\text{trigger}} + B_{\text{router}} + B_{\text{comm}} + B_{\text{audit}} + B_{\text{retry}}
   $$

3. 明确公平性协议：
   - 同样的基础模型
   - 同样的 temperature 范围
   - 同样的 few-shot 规则
   - 同样的外部工具权限
   - 同样的解析失败重试策略

4. 定义主比较口径：
   - equal-total-token
   - equal-latency
   - equal-number-of-model-calls
     至少满足前两个中的一个，最好两个都报。

5. 冻结论文主 claim：
   不是“我们更省 token”，而是
   **“在相同预算下，我们只解决高价值分歧，因此比全量 debate 更有效。”**

### 输出物

- `protocol.md`
- `cost_accounting.md`
- `fairness_checklist.md`

### 过线标准

任何一张主表里的数，都必须能追溯到预算定义和日志记录。

------

## Stage 1：独立求解与结构化输出

### 目标

让每个 agent 先独立完成一轮高质量推理，形成后续通信的原材料。

### 必做任务

1. 为每个 agent 统一输出 schema。
   不要只输出自然语言，应强制结构化：

```json
{
  "final_answer": "...",
  "short_reasoning": "...",
  "confidence": 0.0,
  "uncertain_point": "...",
  "key_evidence": "...",
  "error_risk_type": "...",
  "estimated_need_comm": true
}
```

1. 限制 `short_reasoning` 长度。
   这里不是 full CoT；它只是后续 disagreement extraction 的草稿，必须短。
2. 设计 task-specific 输出：
   - 数学：关键中间式、最后答案、最不确定一步
   - QA：候选答案、支撑句、证据缺口
   - 代码：实现思路、潜在 bug 点、最不确定接口
3. 做置信度校准。
   不能直接相信模型原始 confidence；要做 dev-set calibration，至少报 ECE/Brier 或 rank correlation。
4. 统计独立解的 diversity：
   - 不同 agent 初始答案冲突率
   - 冲突样本中的正确率
   - 一致但错误的比例

### 输出物

- `solver_prompt_v1`
- `structured_parser.py`
- `confidence_calibration_report.ipynb`

### 过线标准

- 结构化输出解析成功率高
- confidence 至少有可用排序价值
- 初始多样性足够，不然多 agent 没意义

------

## Stage 2：触发判断

### 目标

判断一个样本是否值得进入通信层。

### 必做任务

1. 把触发器拆成几个独立信号：
   - answer disagreement
   - mean confidence low
   - confidence spread high
   - verifier uncertainty high
   - difficulty estimator high
2. 先做 heuristic trigger，再考虑 learned trigger。
   顶会初版不要一上来就上 learned policy，不然排错太难。
3. 设计 trigger precedence：
   - 先看硬冲突（答案不一致）
   - 再看高不确定（置信低/分散高）
   - 再看 verifier / difficulty estimator
4. 记录每次触发原因。
   日志中必须有：
   - 是否触发
   - 由哪条规则触发
   - 触发时各 agent 的状态
5. 构造 trigger 的监督评估集。
   从 dev 集抽 100–150 个样本，人工标注“通信是否真的可能改变结果”。

### 输出物

- `trigger_rules.yaml`
- `trigger_eval.csv`

### 过线标准

触发器不能“几乎全开”也不能“几乎不开”。
理想状态是：它只把真正难样本送入通信，而不是把简单题也拖进去。

------

## Stage 3：分歧对象抽取

### 目标

把“多个 agent 的输出”压成可被路由和审计的最小单元。

### 这是你论文里最需要做强的一层

建议把 communication object 明确定义为：

```json
{
  "claim_id": "...",
  "claim_text": "...",
  "supporters": ["a1", "a3"],
  "opposers": ["a2"],
  "confidence_stats": {...},
  "key_evidence_spans": [...],
  "uncertain_spans": [...],
  "conflict_type": "answer|step|evidence|assumption",
  "estimated_gain": ...,
  "estimated_cost": ...
}
```

### 必做任务

1. 定义 conflict taxonomy：
   - answer-level conflict
   - intermediate-step conflict
   - evidence conflict
   - missing-assumption conflict
2. 设计 extraction 规则：
   - 数学：提取关键中间步骤差异
   - QA：提取支持句差异
   - 代码：提取函数行为/边界条件差异
3. 做一小批人工标注，验证 extractor 是否抽到了“真正关键的分歧”。
4. 对 extractor 评估：
   - precision
   - coverage
   - 是否能覆盖真正 outcome-changing 的分歧

### 输出物

- `disagreement_schema.json`
- `annotation_guideline.md`
- `extractor_v1.py`

### 过线标准

被抽出来的分歧，必须足够短、足够关键、可被后续 auditor 消化。

------

## Stage 4：预算感知路由

### 目标

在预算约束下，只挑最值得交流或审计的分歧。

### 必做任务

1. 定义价值函数。
   可以从下面几项加权开始：
   - 冲突强度
   - 置信分散度
   - 证据不重合度
   - verifier uncertainty
   - 任务难度
   - 单位 token 预期收益
2. 定义 cost 函数：
   - 消息长度成本
   - 接收者更新成本
   - auditor 审计成本
   - 额外轮次成本
3. 先做 heuristic router：
   $$
   \max \sum Value(d), \quad s.t.\ \sum Cost(d) \le B
   $$
4. 再决定是否需要 learned router。
   顶会 first version 完全可以只用 heuristic，只要机制清楚、实验扎实。
5. 路由决策必须落日志：
   - 为什么选这个分歧
   - 为什么没选另一个
   - 预算剩余多少

### 输出物

- `router_v1.py`
- `value_cost_ablation.csv`

### 过线标准

你选中的前几个 disagreement objects，应该覆盖大部分真正会改变结果的点。

------

## Stage 5：选择性通信

### 目标

决定“传什么”“传给谁”“传几轮”。

### 必做任务

1. 设计通信内容层级：
   - answer-only
   - answer + confidence
   - disagreement-step only
   - key-evidence only
   - adaptive content
2. 设计通信拓扑：
   - pairwise challenger
   - star/controller
   - selective routing
     不建议默认 all-to-all。
3. 设计 belief update 机制：
   每个接收者收到信息后，必须显式输出：
   - 是否改变答案
   - 置信度变化
   - 改变依据是什么
4. 防止“盲从多数”：
   - 不允许只写“我同意 A”
   - 必须写“我改变/不改变的具体原因”
5. 控制轮次：
   - 首选最多 1 轮
   - 第二轮只给极少数 hard cases
   - 不建议把多轮 debate 当默认路径

### 输出物

- `comm_templates.md`
- `belief_update_parser.py`
- `topology_config.yaml`

### 过线标准

通信必须带来信息增益，而不是把相同内容重复传播。

------

## Stage 6：局部审计

### 目标

只审关键分歧点，而不是重判整道题。

### 必做任务

1. 明确 auditor 的输入边界：
   auditor 只能看：
   - 当前 disputed claim
   - 相关证据
   - 必要上下文
     不能看整段长 debate，否则“局部审计”就失真了。
2. 设计 task-specific 审计方式：
   - 数学：重算关键一步
   - QA：比对 supporting facts
   - 代码：执行单元测试/静态检查
   - 常识逻辑：要求证据链对齐
3. auditor 允许 abstain。
   非常重要。否则 auditor 会被迫胡判。
4. 记录 auditor 行为：
   - resolve / abstain / wrong-overrule
   - minority rescue 是否发生
   - audit cost
5. 明确 fallback：
   - auditor 明确判定 → 采用审计结果
   - auditor abstain → 回退到聚合器
   - 审计失败 → 回退到最强无通信基线

### 输出物

- `auditor_prompt_v1`
- `audit_log_schema.json`

### 过线标准

local auditing 的平均成本必须显著低于 full debate 的额外成本，否则你这层没有意义。

------

## Stage 7：最终聚合与失败保护

### 目标

将独立求解、选择性通信、局部审计的结果整合成最终答案。

### 必做任务

1. 设计聚合优先级：
   - 若无触发：直接 early exit
   - 若有审计且审计成功：优先审计结果
   - 若审计 abstain：走 calibrated vote / weighted aggregation
   - 若系统异常：回退 majority vote
2. 设计最终 confidence：
   不能直接用某个 agent 的 confidence，要综合：
   - 原始 solver confidence
   - 通信后更新幅度
   - auditor verdict
3. 增加 failure-safe：
   - parse error
   - empty evidence
   - conflicting auditor output
   - budget exhaustion
4. 保存完整 trace，用于后续 error analysis 和论文案例图。

### 输出物

- `aggregator.py`
- `fallback_policy.md`

### 过线标准

任何样本都必须有可回放 trace，且失败时有确定的回退路径。

# 四、项目实施层面的详细计划（26 周版）

下面这部分是“怎么把上面的方法流程做成一篇能投顶会/顶刊的论文”。

------

## Phase 0（第 1–2 周）：研究规格冻结

### 任务

第 1 周只做三件事：

1. 冻结问题定义
2. 冻结预算口径
3. 冻结主 baseline 清单

第 2 周做：

1. 写 novelty matrix
2. 定 benchmark 组合
3. 定 equal-budget 协议

### 产出

- 2 页 research spec
- novelty matrix
- experiment protocol

### 里程碑

到第 2 周末，**不再改题目主张**。

------

## Phase 1（第 3–5 周）：基础设施与数据管线

### 任务

第 3 周：

- 数据下载与清洗
- 统一 answer parser
- 统一 scorer
- 跑通单样本日志

第 4 周：

- 实现 cache
- 实现 retry
- 实现 token 统计
- 实现 wall-clock 统计

第 5 周：

- 建 dashboard
- 自动出 csv / plot
- 检查不同实验配置能否完全复现

### 产出

- 可复现实验仓库
- 统一日志格式
- 自动绘图脚本

### 里程碑

任何实验必须做到“一键复跑”。

------

## Phase 2（第 6–8 周）：强基线与 headroom 分析

### 任务

第 6 周：

- Single-agent CoT
- Self-consistency
- Majority Vote

第 7 周：

- Vanilla MAD
- 角色分工基线
- 至少 1 个预算感知/压缩基线近似复现

第 8 周：

- 跑 pilot 子集
- 做错误分类
- 看 headroom 是否存在

### 必做分析

把错误分成：

1. 一致但错
2. 冲突且可修
3. 冲突但噪声传播
4. 高置信错误
5. 投票成功但 debate 无增益

### 里程碑

如果到第 8 周发现 equal-budget 下 communication 完全不优于强无通信基线，就要立刻 pivot。

------

## Phase 3（第 9–11 周）：独立求解器与置信度建模

### 任务

第 9 周：

- 实现 structured output
- 做 parser
- 限制 short reasoning 长度

第 10 周：

- 做 confidence calibration
- 看 confidence 与 correctness 的关系

第 11 周：

- 加 uncertain point / key evidence 输出
- 做 diversity 统计

### 里程碑

独立求解层必须提供“可解释、可抽取、可校准”的输出。

------

## Phase 4（第 12–14 周）：触发器、分歧对象、路由器

### 任务

第 12 周：

- disagreement trigger
- confidence trigger
- verifier trigger
- hybrid trigger

第 13 周：

- 标注 100–150 个样本的关键分歧点
- 训练或规则化 extractor

第 14 周：

- 实现 budgeted router
- 做 top-k disagreement selection

### 里程碑

这阶段结束后，你应该已经能回答一句话：
**“我们为什么挑这些分歧，而不是那些分歧？”**

------

## Phase 5（第 15–17 周）：选择性通信与局部审计

### 任务

第 15 周：

- answer-only / evidence-only / disagreement-only / adaptive 通信模板
- 设计 belief update

第 16 周：

- 实现 local auditor
- 加 abstain
- 加 fallback

第 17 周：

- 实现 end-to-end v1
- 跑第一版完整系统

### 里程碑

到第 17 周末，必须出现第一张可以放论文里的主表雏形。

------

## Phase 6（第 18–21 周）：主 benchmark 扩展与 communication-necessary setting

### 任务

第 18 周：

- full-view benchmark 全量跑

第 19 周：

- 补 communication-necessary setting
  最推荐：split-context HotpotQA
  因为 HotpotQA 自带 supporting facts，天然适合做“局部证据交换与审计”。([HotpotQA](https://hotpotqa.github.io/))

第 20 周：

- 增加一个更硬的 final benchmark
  MMLU-Pro 是很合适的，因为它比 MMLU 更难、更偏 reasoning、对 prompt 更稳定。([arXiv](https://arxiv.org/abs/2406.01574))

第 21 周：

- 如果资源允许，再补一个 coordination/topology benchmark 小规模实验
  AgentsNet 或 MultiAgentBench 可以作为“通信结构确实重要”的额外证据。([OpenReview](https://openreview.net/forum?id=gsSIH0mZ0Y))

### 里程碑

主结果不能只停留在“普通 QA/math 开发集上有效”。

------

## Phase 7（第 22–23 周）：ablation 与机制分析

### 任务

第 22 周：

- trigger ablation
- content ablation
- topology ablation
- agent number ablation
- rounds ablation
- homogenous vs heterogeneous ablation

第 23 周：

- counterfactual analysis
- leave-one-message-out
- leave-one-audit-out
- failure case taxonomy
- confabulation consensus analysis

### 必须画出的图

1. accuracy–budget frontier
2. no-comm / comm / audit 的比例图
3. disagreement value 分布图
4. 成功案例图
5. 失败案例图

### 里程碑

这一阶段结束后，你的论文不只是“结果表”，而是有“机制证据链”。

------

## Phase 8（第 24–26 周）：写作、内审、投稿打磨

### 任务

第 24 周：

- 完成初稿
- 所有表和图冻结编号

第 25 周：

- 做 3 轮 mock review
  一轮挑 novelty
  一轮挑 fairness
  一轮挑实验统计与局限性

第 26 周：

- 修 rebuttal-friendly 版本
- 补 appendix
- 补 reproducibility checklist
- 补代码发布清单

### 里程碑

提交前必须做到：

- 每个 claim 都有实验支持
- 每个 ablation 都能解释主方法
- 每个负结果都有局限性说明
