# 方法流程 v3：面向顶会的正式方法与实验协议

## 0. 文档定位

这份文档不是一般的“方法想法整理”，而是面向顶会投稿的**正式方法流程与实验协议**。

它要解决的是：

- 方法怎么定义才严谨；
- 实验怎么做才不会被 equal-budget 审稿意见一票否决；
- 当前项目里已有的不同 family，如何被统一组织到一个自洽的研究框架下；
- 最终论文应该如何把 same-context 与 split-context 两类 setting 分开、又统一地讲清楚。

## 1. 统一问题定义

## 1.1 论文总问题

我们关心的不是“多智能体是否比单智能体更强”，而是：

**在固定总预算下，多智能体系统如何通过选择性通信与局部验证，尽量保持或提升解题能力。**

这里的“总预算”必须是显式定义的。

## 1.2 总成本定义

建议把总成本统一定义为：
$$
\text{TotalCost}
=
\text{SolveCost}
+
\text{TriggerCost}
+
\text{RoutingCost}
+
\text{CommCost}
+
\text{AuditCost}
$$
其中：

- `SolveCost`
  agent 独立求解成本
- `TriggerCost`
  判断是否需要通信的成本
- `RoutingCost`
  构造冲突对象、选路由、做预算决策的成本
- `CommCost`
  实际消息发送与 belief update 成本
- `AuditCost`
  局部审计或裁决成本

顶会写法里，必须明确：

- 这些成本都计入总 token / 总调用预算；
- 不允许把 router、auditor、verifier 当成“免费模块”。

## 1.3 两类 setting 必须显式分开

为了保证论证自洽，全文必须明确区分：

### A. Same-Context

每个 agent 在初始时看到基本相同的问题上下文。

这一 setting 下的核心问题是：

- 是否有必要通信
- 哪些样本值得通信
- 如何在预算内接近 full communication

### B. Split-Context

每个 agent 在初始时只看到局部信息或证据子集。

这一 setting 下的核心问题是：

- 通信是否构成能力恢复的必要条件
- 传什么信息最有恢复价值
- 在预算内能恢复多少 full-context 性能

如果不把这两类 setting 分开，审稿人很容易指出：

- 你的主张在一类 setting 成立，在另一类并不成立；
- 但你把它们混成一个 overall，削弱了论证有效性。

## 2. 统一框架：SPARC v3

## 2.1 为什么仍建议保留 `SPARC`

从当前项目资产和最新研究看，最适合的不是把论文定位成单一实现 paper，而是：

**把 `SPARC` 定义为一个预算约束下的统一通信框架。**

`SPARC` 的含义可以继续写成：

**Selective Passing and Audited Routing under Communication Budget**

但在 v3 里要明确：

- `SPARC` 是 framework-level 名称；
- 它的不同组件，当前由不同 family 提供最强实证支持；
- `sparc_v1` 是最接近完整表达的一个实例化，而不是 `SPARC` 的唯一实现。

## 2.2 SPARC v3 的四个核心模块

建议将方法正式压缩成 4 个模块：

1. `Independent Solve`
2. `Triggered Communication`
3. `Selective Passing / Budgeted Routing`
4. `Audited Resolution`

这四个模块与当前项目 family 的对应关系如下：

- `selective_comm`
  主要验证 `Triggered Communication`
- `budget_comm`
  主要验证 `Selective Passing / Budgeted Routing`
- `sparc`
  主要验证 `Audited Resolution` 与 message mode 设计
- `comm_necessary`
  在 split-context 下验证通信必要性
- `multi_agent / free_mad_lite / sid_lite / cue`
  作为机制对照、边界情况和 supporting evidence

## 3. SPARC v3 的正式执行流程

## Stage 0：协议冻结

### 目标

在进入任何新一轮大规模实验前，冻结：

- benchmark 集合
- split
- backbone
- cost accounting
- baseline 列表
- acceptance 规则

### 必须冻结的内容

1. `same_context` 主表 benchmark
2. `split_context` 主表 benchmark
3. `best_no_comm_control` 选择规则
4. `full_comm_reference` 选择规则
5. token 统计口径
6. aggregate 与 CI 统计口径

### 当前项目建议

- 开发调试：`smoke20`
- 结构性验证：`pilot100`
- 顶会主表：必须再补一轮更大 confirmatory phase

原因是：

- `pilot100` 已足够判断方向；
- 但对顶会来说，`pilot100` 更像 strong pilot，而不是最终 confirmatory evidence。

## Stage 1：Independent Solve

### 目标

每个 agent 在没有显式通信的前提下独立求解，输出后续所有模块共享的结构化原料。

### 标准输出

建议统一抽象为：

```json
{
  "final_answer": "...",
  "reasoning_sketch": "...",
  "confidence_raw": 0.0,
  "uncertain_point": "...",
  "key_evidence": ["...", "..."],
  "claim_span": "...",
  "error_risk_type": "calculation|retrieval|logic|assumption|unknown"
}
```

当前项目不同 family 的字段名字略有差异，但语义大体落在这些槽位上。

### 顶会要求

- 结构化输出契约必须稳定；
- 不允许在测试阶段继续随意加字段；
- confidence 不能只是“写出来”，必须检查其排序或分箱有效性。

## Stage 2：Triggered Communication

### 目标

决定哪些样本值得进入通信层。

### 允许使用的触发信号

建议只用数据集无关、方法自洽的信号：

- answer disagreement
- mean confidence
- confidence spread
- verifier uncertainty
- lightweight difficulty proxy

### 顶会要求

你必须证明：

- 触发器不是 almost-always-on；
- 也不是 almost-always-off；
- 被触发样本确实更难，或者更可能从 full communication 中受益。

### 当前项目资产映射

- `trigger_early_exit_v1`
  最强的 trigger + early-exit 实证
- `voc_trigger_v2`
  最强的 black-box proxy 触发实证
- `cue_v1`
  最接近统一 utility trigger，但经验上仍偏弱
- `sparc_v1`
  体现 trigger 与 message mode 的联动

## Stage 3：Disagreement Object Construction

### 目标

把 agent 输出压缩成可被通信、路由、审计的最小分歧单元。

### 正式定义

建议统一抽象成：

```json
{
  "conflict_type": "answer|step|evidence|assumption",
  "claim_a": "...",
  "claim_b": "...",
  "support_a": ["..."],
  "support_b": ["..."],
  "confidence_stats": {"mean": 0.0, "spread": 0.0},
  "estimated_value": 0.0,
  "estimated_cost": 0.0
}
```

### 为什么这一层是必须的

因为最新研究已经非常清楚地表明：

- debate 的收益不能简单归因于“让大家多聊几轮”；
- 真正关键的是，是否把高价值冲突从低价值噪声里分离出来。

### 当前项目资产映射

- `cue` 的 `ConflictObject`
- `sparc` 的 disagreement-step / critical-evidence packet
- `comm_necessary` 的 split evidence packet

都能作为这一层的不同实例。

## Stage 4：Selective Passing / Budgeted Routing

### 目标

在预算约束下，只挑少数最值得处理的分歧对象，并为它们分配消息粒度与审计预算。

### 推荐的统一目标函数

\[
S^*=\arg\max_{S\subseteq D,\ \sum cost(d)\le B}\sum value(d)
\]

其中：

- `D`
  当前样本的所有候选分歧对象
- `value(d)`
  修复该分歧的潜在收益
- `cost(d)`
  处理该分歧所需的通信与审计成本

### 顶会要求

你不能只说“我们发更短的消息”。

你必须解释：

- 为什么处理这些分歧而不是那些；
- 为什么把预算花在这里是合理的；
- 这套策略比随机或简单置信启发式更有效。

### 当前项目资产映射

- `budget_comm / dala_lite`
  是当前这一层最强的实证锚点
- `selective_comm`
  更多是“是否触发”，不是完整预算路由
- `sparc`
  更多是消息模式选择与审计，而不是显式 knapsack/auction

## Stage 5：Budgeted Communication

### 目标

决定“传什么、传给谁、传几轮”，且总成本不突破预算。

### 建议的消息粒度层级

从小到大：

1. answer only
2. answer + confidence
3. disagreement-step only
4. critical-evidence only
5. full packet

### 当前项目的强实证

- `comm_necessary`
  提供 answer / evidence / full packet 三种通信强度比较
- `sparc`
  提供 answer_only / answer_confidence / disagreement_step_only / critical_evidence_only / full_cot / task_adaptive
- `sid_lite`
  提供 full vs compressed packet

### 顶会要求

这一层必须展示：

- 内容压缩不是 cosmetic 改动；
- 不同 message mode 的 accuracy-cost tradeoff 可被定量分析；
- message content 真的是决定结果差异的关键轴之一。

## Stage 6：Audited Resolution

### 目标

只对关键分歧做局部验证，而不是把整题交给一个全局 judge 重判。

### 为什么这层非常关键

最新研究里，最接近你当前构想核心的一条线就是：

- `AgentAuditor`
  局部 reasoning-tree auditing 优于 majority vote 和 LLM-as-Judge

因此：

- 如果你的论文要冲顶会，local auditing 不能只做 supporting idea；
- 它应当成为主贡献之一。

### 正式定义

auditor 只能看：

- 问题
- 最小冲突对象
- 相关证据或推导片段
- 必要最小上下文

它不能直接看完整长 debate。

### 推荐输出

```json
{
  "decision": "resolve_for_a|resolve_for_b|abstain",
  "rationale": "...",
  "evidence_used": ["...", "..."],
  "confidence": 0.0
}
```

### 当前项目资产映射

- `sparc_v1`
  最接近“局部审计 + selective passing”的完整实例
- `auditing_ablation`
  用来解释审计层是否真的贡献了收益
- `aggregation_auditing_ablation`
  用来解释审计与聚合是如何耦合的

## Stage 7：Final Aggregation

### 目标

将以下信息统一聚合：

- 独立求解结果
- belief update 后结果
- 局部审计 verdict

### 推荐优先级

1. 若未触发通信：直接 early exit
2. 若触发且审计 resolve：采用 audit result
3. 若审计 abstain：回到 method-defined 聚合器
4. 若结构化失败或预算耗尽：回退 method-defined safe fallback

### 顶会要求

必须记录完整 trace，使得每个样本都能回放：

- 为什么触发
- 抽了哪个分歧
- 发了什么内容
- 审计判了什么
- 最终如何聚合

## 4. 论文主实验协议：必须如何做才严谨

## 4.1 主体比较必须遵守的 6 条规则

### 规则 1：Equal-budget 是硬约束

同一组主结果里，必须显式报告：

- total tokens
- communication tokens
- calls per question

如果可能，还应补：

- wall-clock latency

### 规则 2：matched no-comm controls 是必选项

受 `Debate or Vote` 影响，所有通信类方法都必须与 matched no-comm baseline 比较。

例如：

- `mad_3a_r1` 对 `mv_6 / sc_6`
- `mad_2a_r1` 对 `mv_4 / sc_4`
- `trigger_early_exit` 对 `mv_3 / mv_6 / sc_6`
- `dala_lite` 对 `mv_3`

### 规则 3：same-context 与 split-context 分表

主结果至少要拆成两张表：

- Same-context main table
- Split-context main table

### 规则 4：所有 headline 结果都要带 CI

至少对以下量做 bootstrap CI：

- `faithful_score`
- `delta_vs_best_no_comm`
- `delta_vs_full_comm`

### 规则 5：必须保留负结果

你不能把弱 family 删除得像从未存在过一样。

顶会更看重：

- 你是否清楚哪些方向成立；
- 也是否诚实报告哪些方向当前不够强。

### 规则 6：所有方法比较必须 faithful

不能通过给某个 family 加额外 fallback、额外 judge、额外 early-exit 去把它修成另一个方法。

## 5. 顶会级别的 Related Work 写法

## 5.1 不能只按“谁是 baseline”来写

应该按研究问题来组织：

### 方向 A：多智能体 debate 是否本身有效

- `Debate or Vote`
- `Can LLM Agents Really Debate?`

### 方向 B：如何降低 debate 的通信成本

- Sparse topology
- `S²-MAD`
- `SID`
- `DALA`
- `SELENE`
- `MARS`

### 方向 C：如何降低共识压力与 sycophancy

- `CONSENSAGENT`
- `Free-MAD`

### 方向 D：如何改进最终决策

- `AgentAuditor`
- trajectory-based judging

然后再引出：

**现有工作缺少一套同时覆盖 same-context 与 split-context、并把 selective communication、budget routing 和 local auditing 统一到一个框架里的严谨论证。**

## 5.2 相关工作必须区分证据等级

为了论证稳，论文中应主动区分：

- 已发表 / 已接收工作
- arXiv 预印本
- 在审 / withdrawn 的趋势信号

不要把 withdrawn 或在审论文当作唯一证据锚点。

## 6. 顶会级别的贡献写法

我建议把贡献压缩成 4 条。

### 贡献 1

提出一个统一的预算约束多智能体通信框架 `SPARC`，将独立求解、触发式通信、预算路由和局部审计整合到同一 formal setup 中。

### 贡献 2

系统区分 `same_context` 与 `split_context` 两类 setting，说明它们对应不同的研究问题：

- 前者主要考察 budget-preserving communication
- 后者主要考察 communication-necessary recovery

### 贡献 3

通过多个受控 family 实验表明：

- gains 并不来自默认全量 debate；
- 关键在于对高价值分歧的选择、压缩与局部验证。

### 贡献 4

给出一套更严谨的 equal-budget evaluation protocol，避免把额外算力误写成多智能体结构收益。

## 7. 顶会级别还缺什么

如果目标真的是顶会，目前除了 `pilot100` 之外，还建议再补三类东西。

## 7.1 最终 confirmatory phase

`smoke20` 和 `pilot100` 已经足够做方向判断，但还不足以成为顶会最终证据。

建议再补：

- 更大样本的 `main/dev300/full-dev` phase
- 同样按 frozen protocol 跑

目的是把：

- small-sample optimism
- pilot-stage 调参痕迹

尽量压掉。

## 7.2 更正式的统计检验

除了 CI，还建议补：

- paired bootstrap
- McNemar style sample-wise win/loss

尤其对于：

- `trigger_early_exit` vs `always_communicate`
- `dala_lite_same_context` vs `all_to_all_full`
- `full_packet_exchange` vs `split_no_comm`

这些最关键对比。

## 7.3 预算 frontier 图和阶段 ceiling 图

至少要画两类图：

### 图 1：Budget frontier

- x 轴：communication tokens 或 total tokens
- y 轴：faithful score

### 图 2：Stage ceiling gap

展示哪些方法：

- 还存在 faithful 范围内可回收空间；
- 哪些已经接近方法上限。

这会直接帮助你决定：

- 哪些 family 还值得继续打磨；
- 哪些应该忠实保留为边界或弱结果。

## 9. 一句话结论

如果这篇论文要冲顶会，那么方法与实验协议的核心不应再是：

- “如何让多智能体更会 debate”

而应是：

- **如何在固定预算下，只对真正高价值的分歧做选择性通信与最小化局部验证。**

你现有项目其实已经具备这篇论文需要的大部分模块；
下一步不是再发散，而是把它们收束成一套更统一、更分轨、更严谨的正式 protocol。
