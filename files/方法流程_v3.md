# 方法流程 v3：正式方法框架与实验协议

## 1. 问题定义

## 1.1 固定预算多智能体推理

给定问题 `x` 和 agent 集合 `A = {a_1, ..., a_n}`，系统可以执行：

- 独立求解；
- 触发判断；
- 消息路由；
- agent 间通信；
- belief update；
- 局部审计；
- 最终聚合。

总成本定义为：

```text
TotalCost = SolveCost + TriggerCost + RoutingCost + CommCost + AuditCost
```

所有模型调用和 token 都必须计入总成本。

## 1.2 Same-context

每个 agent 起始看到相同或基本相同信息。

研究问题：

- 哪些样本需要通信；
- 通信是否接近 full communication；
- 是否能减少通信成本；
- 是否避免错误改判。

主要指标：

- faithful score；
- delta vs best no-comm；
- delta vs full-comm；
- token ratio vs full-comm；
- communication tokens；
- calls per question。

## 1.3 Split-context

每个 agent 起始只看到局部信息或证据片段。

研究问题：

- 通信是否能恢复信息缺口；
- 传什么信息最有效；
- 预算路由是否接近 full communication；
- 与 full-context single-agent 还有多少差距。

主要指标：

- answer EM / F1；
- support F1；
- joint F1；
- title recall；
- delta vs split no-comm；
- delta vs full-context；
- communication tokens。

## 2. SPARC 框架

建议使用 `SPARC` 作为论文框架名：

> Selective Passing and Audited Routing under Communication Budget

`SPARC` 是 framework，不是单一实验配置。

四个核心模块：

1. Independent Solve
2. Triggered Communication
3. Selective Passing / Budgeted Routing
4. Audited Resolution

当前项目中的实证对应：

| 模块 | 实证来源 | 作用 |
| --- | --- | --- |
| Independent Solve | `single_agent`, shared no-comm controls | 建立无通信参考 |
| Triggered Communication | `trigger_early_exit_v1`, `trigger_voc_v2` | 判断何时通信 |
| Budgeted Routing | `dala_lite_same_context_v1`, `dala_lite_split_context_v1` | 在预算内分配通信 |
| Message Content | `hotpotqa_split_main`, `content_ablation_v1` | 分析传什么 |
| Audited Resolution | `sparc_v1_smoke`, auditing ablations | 局部验证关键分歧 |
| Debate Control | `multi_agent_main`, `free_mad_lite_v1` | 分析 debate 与 vote 的关系 |
| Diagnostic Utility | `cue_v1`, `sid_lite_v1` | 分析统一 utility 和压缩早退边界 |

## 3. 执行流程

## Stage 0：协议冻结

正式实验前冻结：

- model；
- benchmark；
- phase；
- prompt version；
- method roster；
- primary method；
- no-comm control；
- full reference；
- token accounting；
- acceptance gate。

当前 `pilot100` 使用：

- model: `xiaomimimo/mimo-v2.5`
- matrix size: `16`
- harness: `faithful_matrix`

## Stage 1：Independent Solve

每个 agent 或 solver 独立产生初始答案。

典型结构：

```json
{
  "final_answer": "...",
  "reasoning_sketch": "...",
  "confidence": 0.0,
  "uncertain_point": "...",
  "evidence_items": ["...", "..."]
}
```

要求：

- 初始答案必须可独立评分。
- 后续通信和审计必须可追溯到初始答案。
- `single_agent` 只使用 `cot` 和 `sc`。
- `multi_agent` 的无通信对照使用共享 `mv_k` 控制。

## Stage 2：Triggered Communication

判断是否进入通信阶段。

可使用信号：

- answer disagreement；
- confidence；
- confidence spread；
- self-signal；
- value-of-communication proxy。

当前主要证据：

| experiment | method | score | 关键对比 |
| --- | --- | ---: | --- |
| `trigger_early_exit_v1` | `hybrid_trigger` | `0.840` | 与 `always_communicate` 持平，token ratio `0.547` |
| `trigger_voc_v2` | `voc_trigger_v2` | `0.730` | vs `mv_6` `+0.080`，vs `always_communicate` `-0.010` |

解释：

- `trigger_early_exit_v1` 是当前最强 same-context 触发证据。
- `trigger_voc_v2` 是黑盒 proxy trigger 的正证据。

## Stage 3：Conflict / Message Object Construction

将分歧整理为可通信、可路由、可审计的对象。

建议抽象：

```json
{
  "conflict_type": "answer|step|evidence|assumption",
  "claim_a": "...",
  "claim_b": "...",
  "support_a": ["..."],
  "support_b": ["..."],
  "estimated_value": 0.0,
  "estimated_cost": 0.0
}
```

作用：

- 减少冗余通信；
- 定位关键分歧；
- 为预算路由提供候选；
- 为局部审计提供最小上下文。

## Stage 4：Selective Passing / Budgeted Routing

目标是在预算内选择最值得传递的信息。

形式化目标：

```text
S* = argmax sum(value(d)) subject to sum(cost(d)) <= B
```

当前主要证据：

| experiment | method | score | 关键对比 |
| --- | --- | ---: | --- |
| `dala_lite_same_context_v1` | `dala_lite` | `0.710` | vs `mv_3` `+0.113`，vs full `-0.013` |
| `dala_lite_split_context_v1` | `dala_lite` | `0.695` | vs `mv_3` `+0.085`，vs full `-0.055` |

解释：

- same-context 中，预算路由能接近 full communication。
- split-context 中，预算路由能恢复一部分信息缺口。

## Stage 5：Budgeted Communication

决定传什么。

消息粒度从低到高：

1. answer only
2. answer + confidence
3. disagreement step
4. critical evidence
5. full packet

当前主要证据：

| experiment | method | score | 关键对比 |
| --- | --- | ---: | --- |
| `hotpotqa_split_main` | `full_packet_exchange` | `0.610` | vs `split_no_comm_mv3` `+0.210`，vs `full_context_single` `-0.070` |
| `content_ablation_v1` | `task_adaptive` | `0.623` | vs `full_cot` `-0.097` |

解释：

- split-context 中，通信内容决定恢复能力。
- 内容设计错误会明显损害结果。

## Stage 6：Audited Resolution

对关键分歧做局部验证。

审计输入：

- 问题；
- 最小冲突对象；
- 相关证据；
- 必要推理片段。

审计输出：

```json
{
  "decision": "resolve_for_a|resolve_for_b|abstain",
  "rationale": "...",
  "confidence": 0.0
}
```

当前主要证据：

| experiment | method | score | 关键对比 |
| --- | --- | ---: | --- |
| `sparc_v1_smoke` | `sparc_v1` | `0.667` | vs `mv_3` `+0.040`，与 `always_communicate` 持平 |
| `auditing_ablation_v1` | `local_auditing` | `0.667` | vs `majority_vote` `+0.040` |
| `aggregation_auditing_ablation_v1` | `local_auditing` | `0.667` | vs `final_round_vote` `+0.040` |

解释：

- 局部审计有稳定正信号。
- 当前它更适合作为框架组件证据，而不是唯一 headline。

## Stage 7：Final Aggregation

最终聚合遵守方法自身定义。

faithful 要求：

- 不临时新增 fallback；
- 不外接额外 judge；
- 不临时改变通信轮数；
- 不临时改变 agent 数；
- 不使用数据集专用阈值。

## 6. 后续实验

## 6.1 Confirmatory phase

`pilot100` 是强 pilot，不是最终顶会主实验规模。

下一步建议：

- 增加 `main300` 或更大确认集；
- 冻结 method roster；
- 不再按数据集调阈值；
- 分别报告 same-context 与 split-context。

## 6.2 统计显著性

必须补：

- bootstrap confidence interval；
- paired win/loss；
- McNemar-style test；
- dataset-wise breakdown。

优先比较：

- `hybrid_trigger` vs `always_communicate`；
- `hybrid_trigger` vs `sc_6`；
- `voc_trigger_v2` vs `mv_6`；
- `dala_lite_same_context_v1` vs `all_to_all_full`；
- `hotpotqa_split_main` vs `split_no_comm_mv3`；
- `dala_lite_split_context_v1` vs `all_to_all_full`。

## 6.3 Equal-budget single-agent baseline

必须补强：

- budget-matched long CoT；
- budget-matched self-consistency；
- budget-matched full-context single-agent。

目标是验证多智能体收益是否超过同预算单智能体推理。

## 6.4 机制图

至少绘制：

- budget frontier；
- trigger utility curve；
- stage ceiling gap；
- helpful / harmful communication breakdown；
- same-context 与 split-context 分轨图。

## 7. 最终结论

当前方法路线应固定为：

> 固定预算下的高价值分歧识别、通信、路由与局部验证。

这条路线能统一解释当前主要结果：

- `trigger_early_exit_v1` 说明通信需要选择性触发；
- `trigger_voc_v2` 说明黑盒 proxy trigger 有价值；
- `dala_lite` 说明预算路由接近 full communication；
- `hotpotqa_split_main` 说明 split-context 中通信具有必要性；
- `sparc` 说明局部审计是重要组件；
- `cue` 和 `sid_lite` 说明统一估计与压缩早退仍有边界。

因此，论文应写成框架型、分轨评测型、equal-budget 严格控制型工作。
