# 一、正式版总定义

## 1. 论文目标

你的论文不是要证明“多 agent 更强”。

而是要证明：

**在固定总预算下，多智能体推理的核心收益，来自对高价值分歧的识别、路由和局部验证，而不是默认全量 debate。**

## 2. 统一优化目标

从今天开始，所有实验都围绕这一个目标：

[
\max_{\pi}\ \text{Accuracy}(\pi) \quad \text{s.t.} \quad \text{TotalCost}(\pi)\le B
]

其中：

[
\text{TotalCost}= \text{SolveTokens}+\text{TriggerTokens}+\text{RouteTokens}+\text{CommTokens}+\text{AuditTokens}
]

如果你愿意更严格一点，可以另外报告：

- 总 token 成本
- 总模型调用次数
- 平均 wall-clock latency

但**主优化指标必须固定为 equal-total-budget 下的准确率**。

## 3. 论文主假设

你整篇论文只围绕三个假设展开：

H1：简单样本不需要通信，应该 early exit。
H2：困难样本不需要全量 debate，只需要解决少数高价值分歧。
H3：最终提升主要来自局部审计和高质量分歧处理，而不是简单多数投票。

## 4. 正式版方法名

可以继续用你文档里的暂名：

**SPARC**
Selective Passing and Audited Routing under Communication Budget。

## Stage 0：研究协议冻结

### 目标

先把“论文到底比较什么、怎么算成本、怎么保证公平”固定住。

### 你要做的事

1. 冻结任务集合。
   开发阶段保留你文档里的 GSM8K、MATH500、StrategyQA、HotpotQA；正式主表阶段建议再补 MMLU-Pro。HotpotQA 适合做局部证据审计，因为它有 supporting facts supervision；MMLU-Pro 更难、更偏 reasoning，而且对 prompt 更稳，更适合作为硬 benchmark。 ([HotpotQA](https://hotpotqa.github.io/))
2. 冻结基础模型。
   第一版只用**同一个 backbone**。不要一上来混模型。先把方法本身做干净。
3. 冻结 baseline。
   必须至少包含：
   - Single-agent CoT
   - Self-consistency
   - Majority Vote
   - Vanilla MAD
   - DALA / auction-style communication
   - SID
   - AgentAuditor
     这些正好对应当前最容易被问到的对照方向。 ([arXiv](https://arxiv.org/abs/2511.13193))
4. 冻结预算档位。
   建议三档：
   - Low budget
   - Medium budget
   - High budget
     例如分别对应不同总 token 上限。
5. 冻结统计协议。
   每个主结果至少做：
   - 3 次重复
   - 或 bootstrap confidence interval
     不要只报单次最好结果。

### 产出物

- `protocol.md`
- `cost_accounting.md`
- `benchmark_manifest.yaml`
- `baseline_manifest.yaml`

### 验收标准

到这一阶段结束后，你后面**不再随意换 benchmark、换评价口径、换成本定义**。

## Stage 1：独立求解器（Independent Solve）

### 目标

每个 agent 先独立解题，产生后续通信和审计所需的“原材料”。

### 输入

- 问题 `q`
- task type（math / QA / code / commonsense）
- solver prompt

### 输出

每个 agent 必须结构化输出，不准只吐自然语言。建议统一成：

```json
{
  "final_answer": "...",
  "reasoning_sketch": "...",
  "confidence_raw": 0.0,
  "uncertain_point": "...",
  "key_evidence": ["...", "..."],
  "needs_help": true,
  "error_risk_type": "calculation|retrieval|logic|assumption|unknown"
}
```

### 一步步怎么做

1. 写 solver prompt。
   让 agent 完成独立求解，但把 reasoning sketch 控制在固定长度，比如 80–150 tokens。
2. 做 JSON 强约束输出。
   先不要追求最好回答，先追求**解析稳定**。
3. 先抽 50 个样本试跑。
   检查三件事：
   - JSON 是否稳定
   - 答案解析是否正确
   - key evidence / uncertain point 是否有内容
4. 加 repair parser。
   如果 JSON 轻微损坏，自动修复。
5. 做置信度校准。
   不要直接信模型输出的 raw confidence。
   你可以在开发集上做：
   - binning + reliability diagram
   - isotonic calibration
   - rank correlation 检查
6. 统计独立求解多样性。
   你要知道：
   - 初始答案一致率
   - 一致但错误的比例
   - 冲突且可修正的比例
   - 高置信错误比例

### 为什么这层必须强

如果 Stage 1 输出垃圾，后面一切都会失效。
因为 Stage 2 要靠它触发，Stage 3 要靠它抽分歧，Stage 6 要靠它做局部审计。

### 验收标准

- JSON 解析成功率高
- reasoning sketch 够短但还能用
- confidence 至少具备排序意义
- key evidence 不是空的

------

## Stage 2：触发器（Should We Communicate?）

### 目标

决定一个样本是否值得进入通信层。

### 触发信号

我建议一开始不要做 learned trigger，先做 rule-based / heuristic trigger，拆成 5 个独立信号：

1. `answer_disagreement`：多个 agent 最终答案不一致
2. `mean_conf_low`：平均置信度低
3. `conf_spread_high`：置信度分散度高
4. `verifier_uncertain`：轻量 verifier 认为当前答案不稳
5. `difficulty_high`：轻量难度估计器认为题目难

### 一步步怎么做

1. 先定义 5 个指标的计算方法。
   比如：
   - answer_disagreement：答案是否冲突
   - mean_conf：平均 confidence
   - conf_spread：max-min 或方差
   - verifier score：小 verifier 给出的接受概率
   - difficulty：一个小模型或启发式规则
2. 先写 hard rules。
   第一版建议：
   - 若答案不一致，直接触发
   - 若答案一致但平均置信低，触发
   - 若答案一致、平均置信高、spread 小，不触发
   - verifier 明显否定时，触发
3. 建 trigger 验证集。
   从开发集抽 120 个样本，人手标注：
   - 这个样本是否“值得通信”
   - 可能改变结果的关键原因是什么
4. 调阈值。
   目标不是最高 recall，而是**合适的触发率**。
   太高会退化成 always communicate。
   太低会漏掉真正难题。
5. 记录触发原因。
   每个样本必须落日志：
   - 是否触发
   - 因哪条规则触发
   - 触发前各 agent 状态

### 验收标准

- 触发器不能几乎全开
- 也不能几乎全关
- 触发样本应显著比未触发样本更难

### 常见错误

- 把触发器做成“保守全开”
- 置信度完全不校准
- verifier 太重，自己吃掉太多预算

------

## Stage 3：分歧对象抽取（Disagreement Object Construction）

### 目标

把多个 agent 的输出，压缩成若干个“可路由、可审计、可比较”的最小分歧单元。

### 正式定义

建议你把 communication object 明确定义为：

```json
{
  "qid": "...",
  "claim_id": "...",
  "conflict_type": "answer|step|evidence|assumption",
  "claim_text": "...",
  "supporters": ["agent_1", "agent_3"],
  "opposers": ["agent_2"],
  "confidence_stats": {
    "mean": 0.0,
    "spread": 0.0
  },
  "evidence_set": ["...", "..."],
  "uncertain_spans": ["..."],
  "estimated_value": 0.0,
  "estimated_cost": 0.0
}
```

### 一步步怎么做

1. 定义 conflict taxonomy。
   至少拆成 4 类：
   - answer-level
   - intermediate-step
   - evidence-level
   - assumption-level
2. 写 extractor v1。
   先别上复杂模型，先用规则 + 小模型混合：
   - 比较 final_answer
   - 比较 reasoning sketch 中关键步骤
   - 对 QA 任务对齐 key_evidence
   - 对 math 任务抽关键中间式
3. 做人工标注集。
   建议标 100–150 个样本。
   每个样本标：
   - 真正关键分歧点在哪里
   - 是哪一类冲突
   - 如果只解决这一个点，是否可能改变最终答案
4. 评估 extractor。
   不是只看整体准确率，而是看：
   - 抽到的分歧是否真关键
   - 是否覆盖 outcome-changing 分歧
   - 是否过于冗长
5. 限制对象长度。
   每个 disagreement object 必须短。
   不能把整段 CoT 塞进去，不然你只是换了个名字在传 full debate。

### 验收标准

- 抽出的对象短、准、关键
- 能被 auditor 单独消费
- 能区分“答案冲突”和“证据冲突”

------

## Stage 4：预算感知路由（Budgeted Router）

### 目标

在预算约束下，只挑少数最值得交流或审计的分歧。

### 路由目标

[
S^*=\arg\max_{S \subseteq D,\ \sum cost(d)\le B}\sum value(d)
]

其中 `D` 是所有分歧对象。

### value 设计

第一版先用启发式：

[
value(d)=
w_1 \cdot \text{conflict_strength}+
w_2 \cdot \text{conf_spread}+
w_3 \cdot \text{evidence_mismatch}+
w_4 \cdot \text{verifier_uncertainty}-
w_5 \cdot \text{expected_cost}
]

### 一步步怎么做

1. 定义 cost。
   cost 不能只算消息 token。至少要包括：
   - 发送成本
   - 接收者更新成本
   - auditor 审计成本
   - 可能的第二轮成本
2. 实现 top-k / greedy-knapsack。
   第一版不要太复杂，直接按 `value / cost` 排序即可。
3. 决定路由对象。
   这一步输出的不是“开始聊天”，而是：
   - 哪个 disagreement 要处理
   - 给谁处理
   - 用哪种消息粒度处理
   - 预留多少预算给 auditor
4. 记录未被选中对象。
   很重要。后面写论文时，你要能分析“为什么没选它”。
5. 预留 audit budget。
   不要把预算全花在通信上。
   你这篇论文的关键不是 debate，而是最后的局部核查。

### 验收标准

- 路由后的分歧数量明显少于原始分歧数量
- 被选中的对象显著更可能改变答案
- 总成本不爆预算

------

## Stage 5：选择性通信（Selective Communication）

### 目标

决定“传什么、传给谁、传几轮”。

### 消息粒度建议

从小到大设计 5 档：

1. answer only
2. answer + confidence
3. disagreement-step only
4. critical-evidence only
5. adaptive content

### 第一版默认策略

- 默认只 1 轮通信
- 默认只发 disagreement object 的压缩版
- 只有极难样本才允许第二轮

### 一步步怎么做

1. 设计通信模板。
   每条消息必须是结构化的，建议：

```json
{
  "sender": "agent_2",
  "recipient": "agent_1",
  "conflict_type": "step",
  "my_claim": "...",
  "why_i_disagree": "...",
  "key_evidence": ["..."],
  "confidence": 0.61,
  "request": "revise_or_rebut"
}
```

1. 限制长度。
   每条消息设硬 token cap。
2. 实现 belief update。
   接收者收到消息后，不是自由发挥，而是必须输出：

```json
{
  "changed_answer": true,
  "new_answer": "...",
  "confidence_delta": -0.12,
  "reason_for_change": "...",
  "remaining_disagreement": "..."
}
```

1. 不允许“空同意”。
   不能只写“我同意 agent_1”。
   必须写具体改变原因。
2. 决定第二轮规则。
   只有满足以下条件之一，才允许第二轮：
   - 第一轮后仍 unresolved
   - auditor 认为证据不足
   - 当前样本还在预算内

### 验收标准

- 通信后总 token 明显低于 full CoT exchange
- 不是所有样本都进入第二轮
- belief update 能看出真实信息增益

------

## Stage 6：局部审计（Local Auditing）

### 目标

只审关键分歧点，不重判整道题。

### 这是论文最重要的一层

因为最近这条线上，一个最关键的改进点就是：不要把最终裁决做成简单 majority vote，而是把决策缩到 reasoning tree 的关键分歧点上。([arXiv](https://arxiv.org/abs/2602.09341))

### auditor 输入必须受限

auditor 只能看：

- 题目
- 当前 disagreement object
- 相关证据
- 必要的最小上下文

不能给 auditor 看完整长 debate。
否则你这层就不是 local auditing，而是另一种 global judge。

### 一步步怎么做

1. 设计 auditor 三态输出。
   auditor 不能只有 win/lose，必须允许：
   - `resolve_for_A`
   - `resolve_for_B`
   - `abstain`
2. 按任务做局部验证。
   - math：重算冲突那一步
   - QA：核查 supporting evidence 是否真支撑答案
   - code：跑针对该分歧的最小测试
   - commonsense：核查假设是否自洽
3. 限制 auditor 输入长度。
   给 auditor 的上下文越小，越能证明你是在做局部验证而不是全局重判。
4. 落完整日志。
   至少记录：
   - auditor 是否 resolve
   - resolve 给了谁
   - 是否 abstain
   - 是否出现 wrong overrule
5. 定 fallback。
   若 auditor abstain，则回退到聚合器。
   不要强行让 auditor 每次都判。

### 验收标准

- auditor 平均成本明显低于 full judge / full debate
- 能解决一部分多数投票会犯错的样本
- abstain 率合理，不是假装全知

------

## Stage 7：最终聚合与失败保护（Aggregation + Fallback）

### 目标

把独立求解、通信更新、局部审计的结果整合成最终答案。

### 推荐优先级

1. 若未触发通信：直接 early exit
2. 若已审计且审计 resolve：采用审计结果
3. 若审计 abstain：做 calibrated weighted vote
4. 若解析失败 / 预算耗尽：回退最强无通信基线

### 一步步怎么做

1. 写 aggregator 规则。
   不要用自由 prompt，总结为明确 if-else 逻辑。
2. 设计最终置信度。
   综合以下信息：
   - 原始 solver confidence
   - 通信后的变化幅度
   - auditor verdict
3. 做失败保护。
   必须有：
   - parse fail fallback
   - empty evidence fallback
   - budget exhaustion fallback
   - contradictory audit fallback
4. 存完整 trace。
   每个样本都能回放：
   - 谁先答
   - 为什么触发
   - 抽了哪些分歧
   - 路由给了谁
   - 审计如何判
   - 最终如何聚合

### 验收标准

- 每个样本都有完整 trace
- 所有失败都有确定 fallback
- 聚合规则足够稳定，不依赖临场 prompt 运气

# 三、贯穿全程的工程 SOP

这部分最重要，因为它决定你后面能不能复现、能不能写出严谨的实验部分。

## 1. 代码仓库结构

建议一开始就按下面建：

```text
project/
  configs/
    datasets/
    models/
    methods/
  data/
    raw/
    processed/
    splits/
  prompts/
    solver/
    verifier/
    communicator/
    auditor/
  src/
    dataset/
    solver/
    trigger/
    extractor/
    router/
    comm/
    auditor/
    aggregator/
    metrics/
  local/runs/
    exp_****/
  reports/
    tables/
    figures/
    error_cases/
  paper/
```

## 2. 单个实验的标准执行流程

每跑一个实验，严格按这 10 步走：

1. 写 `experiment_card.md`
   说明这次实验比什么、预算多少、模型是谁、预期回答什么问题。
2. 生成 config
   包括：
   - dataset
   - split
   - model
   - seeds
   - budget
   - method variant
3. 先跑 20 个样本 dry-run
   检查：
   - JSON 是否可解析
   - token 统计是否完整
   - 日志字段是否全
4. 再跑 200 个样本 pilot
   看：
   - 触发率
   - early exit 率
   - 平均 token
   - 是否有明显 bug
5. 看错误样本
   人工抽样 20–30 个 case，检查：
   - 是 trigger 错了
   - extractor 错了
   - router 错了
   - auditor 错了
   - aggregator 错了
6. 修规则或阈值
   这一步只允许改开发集，不许碰测试集。
7. 跑 full evaluation
   正式出结果。
8. 统计置信区间
   至少 bootstrap。
9. 更新主结果表和 frontier 图。
10. 归档
    保存：

- config
- code commit
- prompt hash
- 原始日志
- 表格输出

## 3. 每周固定节奏

建议你每周按这个节奏推进：

周一：确定本周唯一主问题
周二：dry-run + pilot
周三：full run
周四：error analysis + ablation
周五：总结、更新图表、写周报

不要一周同时改 4 个模块。
每周只追一个主要问题。

## 4. 三条硬纪律

第一，**任何主结果都必须是 equal-budget 比较**。
第二，**任何方法改动都必须单独登记到实验卡**。
第三，**任何“看起来提升很大”的结果都必须先排查是不是成本漏算了**。

------

# 四、28 周正式计划安排

我建议用 **28 周**。原因很简单：
你不仅要做方法，还要做强 baseline、误差分析、机制分析、图表、写作和内审。26 周也能做，但 28 周更稳，留出了论文化和返工空间。

------

## Phase A：第 1–2 周

## 主题：立项冻结

### 第 1 周

你只做 4 件事：

1. 写正式版 research spec
   内容必须包括：
   - 问题定义
   - 主假设
   - 成本定义
   - 主指标
   - 目标 benchmark
   - baseline 清单
2. 建仓库和目录
   先不要跑实验，先把结构搭好。
3. 冻结数据集
   开发：GSM8K / MATH500 / StrategyQA / HotpotQA
   扩展：MMLU-Pro。 ([HotpotQA](https://hotpotqa.github.io/))
4. 冻结方法结构图
   把原来的 3 阶段图改成 8 阶段图。

### 第 2 周

1. 写 baseline manifest
2. 写 cost accounting 文档
3. 定三档预算
4. 写 evaluation protocol
5. 做第一个 dry-run dataset split

### 第 2 周末必须产出

- 一份 2–3 页 protocol
- 一张系统流程图
- 一个空仓库骨架
- 一份 baseline 对照表

### 本阶段硬门槛

**第 2 周后不再改题目主张。**

**equal-budget 下，多数投票已经吃掉大部分收益，而你的目标样本太少**，那就要及时调整问题定义，而不是硬堆系统。这个判断和最近对 MAD 的反思是一致的。([NeurIPS](https://neurips.cc/virtual/2025/poster/116557))

# 七、三次必须停下来复盘的节点

## 节点 1：第 5 周

问题：
无通信基线有多强？
如果多数投票已经非常强，你后面就不能把论文写成“通信天然有效”。

## 节点 2：第 8 周

问题：
是否真的存在足够多“值得通信”的样本？
如果没有，你要调整问题定义，不要继续堆模块。最近 Debate-or-Vote 的结果本身就在提醒这一点。([NeurIPS](https://neurips.cc/virtual/2025/poster/116557))

## 节点 3：第 20 周

问题：
端到端 SPARC 是否在 equal-budget 下打赢一个强基线？
如果还没有，优先回头修 trigger / router / audit，而不是盲目扩 benchmark。

# 八、最容易犯的 8 个错误

1. 把成本只算通信 token，不算 controller / auditor。
2. 让 trigger 退化成 almost always on。
3. disagreement object 太长，本质上又在传 full CoT。
4. router 没有 reserve audit budget。
5. auditor 看了整段上下文，失去“局部审计”意义。
6. 测试集上调阈值。
7. 只报最优点，不报整个 budget frontier。
8. 主结果只比 Vanilla MAD，不比 Majority Vote。

------

# 九、最后给你的执行建议

真正开始做时，你不要把它理解成“我要同时做一个多 agent 系统”。

你应该把它理解成四个依次推进的小问题：

第一步，**先把无通信下界做实**。
第二步，**再判断哪些样本值得通信**。
第三步，**再把值得通信的内容压成分歧对象**。
第四步，**最后才是通信和局部审计**。

这个顺序不能反。
因为如果前两步没站稳，后面的系统复杂度只会掩盖问题，不会解决问题。

如果你愿意，我下一条可以直接继续给你一份：
**“论文方法部分的正式写法模板（Problem Setup / Method / Algorithm / Training-Free Implementation / Complexity / Ablation Questions）”**，你可以直接拿去写论文初稿。
